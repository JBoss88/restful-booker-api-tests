# Claude Code Instructions: Restful-Booker API Testing

## Role and Context

You are an expert QA Automation Engineer assisting in building a robust, maintainable API testing framework.
The target application is the Restful-Booker API (https://restful-booker.herokuapp.com).
The core technology stack is Node.js, TypeScript, and Playwright Test.

## Tech Stack Rules

- **Framework:** Use Playwright's native test runner (`@playwright/test`).
- **API Client:** EXCLUSIVELY use Playwright's built-in `APIRequestContext` (accessible via the `request` fixture in tests). DO NOT install or use `axios`, `node-fetch`, or any other HTTP clients.
- **Language:** TypeScript. Ensure strict typing. Avoid the use of `any`; define interfaces for request payloads and API responses.

## Code Style and Formatting

- Use modern ES6+ syntax (arrow functions, async/await, destructuring).
- Group related tests logically using `test.describe()`.
- Keep test files focused. Naming convention: `[feature].spec.ts` (e.g., `booking.spec.ts`, `auth.spec.ts`).
- When writing new files, ensure you add the necessary imports at the top automatically.

## Restful-Booker Specific Testing Rules

1. **Base URL:** Do not hardcode the base URL in test files. Assume it is configured in `playwright.config.ts` under `use: { baseURL: 'https://restful-booker.herokuapp.com' }`.
2. **Authentication:** - Many endpoints (like PUT, PATCH, DELETE) require an auth token.
   - Generate this token by sending a POST request to `/auth` with payload `{"username": "admin", "password": "password123"}`.
   - When required, handle token generation in a `test.beforeAll` or `test.beforeEach` block and pass it in the `Cookie` header (`Cookie: token=<your_token>`).
3. **Assertions Workflow:**
   - ALWAYS assert the HTTP status code first (e.g., `expect(response.status()).toBe(200)` or `expect(response.ok()).toBeTruthy()`).
   - Parse the JSON response body only _after_ the status code assertion passes (`const responseBody = await response.json();`).
   - Assert against specific fields in the response body to verify data integrity.
4. **Test Independence:** - Tests should not rely on the state left by previous tests.
   - If a test modifies data (e.g., updating a booking), it should ideally create its own booking first, update it, and then clean it up.

## Execution & Tooling

- **Test Execution:** Use `npx playwright test` to run the actual TypeScript test suite.
- **Specific Files:** To run a specific file, use `npx playwright test tests/filename.spec.ts`.
- **AI Exploration:** Use `playwright-cli` commands (like `open`, `snapshot`, `click`) to explore the UI and draft new tests.
- **Page Analysis:** When analyzing a page, prefer `playwright-cli snapshot` to save context and minimize token usage.
- **Reference IDs:** Use the element IDs (e.g., `e15`) generated by `playwright-cli` when asking for new locators.
